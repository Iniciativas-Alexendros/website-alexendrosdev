## El problema: secretos por todas partes

Cuando trabajas con muchos proveedores externos —Notion, Stripe, GitHub, Vercel, Dokploy, n8n…— los secretos se esparcen. Una variable de entorno aquí, un `.env` allá, un token pegado en la configuración de un servicio que nadie recuerda haber tocado. Cada copia es una superficie de ataque y un punto ciego de auditoría.

> Si no queda registro de quién accedió a un secreto y cuándo, no ha pasado. Y si no ha pasado, no puedes razonar sobre tu seguridad.

TrenchPass nace de esa incomodidad: **un único custodio de credenciales para todo el ecosistema**, en lugar de veinte custodios improvisados.

## La idea: un gateway MCP

La interfaz es el [Model Context Protocol](https://modelcontextprotocol.io). En vez de repartir tokens, los consumidores piden el secreto que necesitan a un gateway que los expone como _tools_ MCP. El gateway —escrito en **Rust** con `axum` y `rmcp`— habla con HashiCorp Vault (almacén caliente, con caché de 60 s) y registra cada acceso.

## Doble factor de verdad

El acceso al gateway exige **dos factores simultáneos**:

- **mTLS**: certificado mutuo, así que el cliente demuestra quién es a nivel de transporte.
- **Bearer token**: autorización a nivel de aplicación.

Sumado a rate limiting y protección anti-replay, la barrera de entrada deja de ser «tener el token» para ser «ser el cliente correcto, con el certificado correcto, dentro de la ventana correcta».

## Auditoría que no se puede borrar

Cada evento va a una tabla **append-only** en PostgreSQL: no se actualiza ni se borra, solo se añade. La telemetría (trazas y métricas) se exporta con **OpenTelemetry** a **SigNoz**, así que la pregunta «¿quién pidió la clave de Stripe el martes?» tiene respuesta sin arqueología.

## Estado

TrenchPass es **open source (AGPL-3.0)** y está en desarrollo activo: el scaffold compila y el enrutado de tools se construye namespace a namespace. Es, ante todo, una afirmación de principios: **la custodia de secretos se diseña, no se improvisa**.
